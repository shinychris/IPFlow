# AI痕迹去除策略指南

软著申请材料去AI化策略参考。将AI辅助生成的源代码与用户手册改造为具有人类编写特征的最终材料。

> **使用原则**：去AI化是让AI辅助材料看起来更像人工编写，**不是伪造材料**。软件本身必须真实存在且可运行，材料内容必须如实反映软件功能。

---

## 一、为什么需要去AI化

AI生成内容存在统计可识别特征，主流检测系统基于以下三大技术路径：

1. **统计特征分类器**（最成熟）：分析文本困惑度、突发性、词汇多样性等维度
2. **水印技术**：部分AI服务在生成内容时嵌入特定模式
3. **深度学习端到端模型**：使用专门神经网络识别更细微的模式差异

理解检测原理是去AI化的前提——只有知道"检测器在看什么"，才能有针对性地"隐藏"AI痕迹。

### 关键检测指标

| 指标 | AI生成 | 人类写作 | 软著材料硬性目标 | 检测敏感度 |
|------|--------|----------|------------------|------------|
| 困惑度（Perplexity） | 12-15（低） | 25-35（高） | 接近 25+ | 极高 |
| 突发性（Burstiness） | 0.3-0.4（低） | 0.65-0.85（高） | 接近 0.65+ | 极高 |
| **代码注释覆盖率** | 80-90%（高） | 20-40%（中） | **<5%（极低）** | 极高 |
| 连接词密度 | ~18%（高） | ~9%（低） | <9% | 高 |
| 词汇多样性（TTR） | ~0.42（低） | ~0.68（高） | 接近 0.68 | 高 |
| 句子长度方差 | 小（均匀） | 大（波动） | 大（波动） | 中高 |
| 段落长度一致性 | >90%（高度一致） | ~60%（随机） | ~60% | 中等 |

## 二、AI生成内容的可识别特征

### 文本内容的AI特征

1. **过度使用连接词**："因此"、"此外"、"值得注意的是"、"综上所述"、"显而易见"——出现频率是人类写作的2-3倍
2. **句式过于规整**：句子长度均匀，缺乏极短/极长句
3. **词汇多样性不足**：TTR指数低于人类写作约20-30%，倾向使用"安全"词汇
4. **情感中性化**：缺乏激情、讽刺、幽默色彩
5. **段落模板化**：常采用"总-分-总"固定结构，常以"随着...的发展"开头，以"综上所述"结尾
6. **信息密度均匀**：每句承载信息量均匀，缺乏人类写作的"信息爆发"

### 代码内容的AI特征

1. **注释覆盖率过高**：80-90%且格式统一（JSDoc/JavaDoc风格）—— 这是 AI 代码**最强指纹**，软著申请材料**硬性要求注释率 <5%**
2. **变量命名过于完美**：完整、语义清晰、符合规范（如 `calculateAnnualInterest`、`userAuthenticationToken`）
3. **代码结构教科书式**：函数长度均匀（20-40行）、圈复杂度低（3-8）
4. **风格一致性过高**：>95%，而人类代码仅50-70%
5. **错误处理覆盖率过高**：>95%，而人类代码60-80%
6. **缺少个人风格标记**：无TODO/FIXME/个人备注

### 软著申请材料的特殊风险点

- **格式过于完美**：排版、格式规范得反而显得不自然
- **内容一致性异常**：源代码与文档关联过于"完美"
- **功能性注释过多**：注释中重复代码显而易见的逻辑
- **用户手册过于专业**：缺少真实文档中的不完整说明、待补充标记

## 三、源代码去AI化策略

### 3.1 注释的人为化处理（⚠️ 硬性指标：注释率 <5%）

注释是AI生成代码最显著的"指纹"。**软著申请材料的源代码注释覆盖率必须 <5%**（即每 100 行代码中注释行不超过 5 行）。核心操作：

- **削减注释 95% 以上**：删除一切显而易见的注释，包括但不限于：
  - ❌ `i++ // 自增计数器`
  - ❌ `// 用户登录方法` + 紧跟的 `public User login(...)`
  - ❌ `// 检查用户名是否存在` + 紧跟的 `if (userRepository.existsByUsername(...))`
  - ❌ 整段的 JSDoc/JavaDoc 块注释（`/** ... */`），尤其是带 `@param` `@return` 的
  - ❌ 类级别的说明注释（`/** 用户服务类，处理用户相关业务逻辑 */`）
- **仅保留以下三类注释**（且总量 <5%）：
  1. 复杂算法的非显然解释（如排序策略选择、性能优化的 trick）
  2. 关键业务规则的固化（如 `// 金额必须保留2位小数，否则对账失败`）
  3. 防止后续误改的"陷阱注释"（如 `// 不要改这里的顺序，依赖数据库索引`）
- **混合多种注释风格（打破模板感）**：
  - 有些函数用单行 `//`，有些用 `#`（Python），有些干脆不写
  - **严禁**全项目统一使用 JSDoc/JavaDoc 风格 —— 这是 AI 代码最强指纹
- **加入 TODO/FIXME/HACK 标记**：如 `// TODO: 这里需要优化，先临时这样处理`
- **添加个人化注释**（控制在 5% 以内）：
  - `// 这个逻辑有点绕，但实测性能最好`
  - `// 这里用了一个trick，后续要重构`
  - `// 客户的需求变更，临时加的`
- **核查方法**：使用 `python scripts/ai_trace_analyzer.py --type code --path <文件>` 检查，注释覆盖率必须显示 <5% 才算达标

### 3.2 变量命名不完美化

- **混合缩写和全称**：
  - `userAuthenticationToken` → `uauthToken` / `authTok`
  - `calculateAnnualInterest` → `calcInterest` / `calcAnnInterest`
- **引入临时变量**：`temp`、`tmp`、`x`、`y`、`flag`，特别是在处理中间结果时
- **使用个人习惯命名**：下划线前缀表示私有（`_userName`），特定前缀（`m_`、`s_`）
- **风格不一致**：同一份代码中部分变量用 camelCase，部分用 snake_case，模拟"不同时间段、不同心情下写的代码"

### 3.3 代码结构真实化

- **调整函数长度**：将过于清晰的小函数合并为50-100行的大函数（"懒得拆"），同时将完美的长函数拆成几个不均匀的短函数
- **引入重复代码**：在2-3处故意保留少量重复代码（如输入验证逻辑），不提取公共函数
- **增加圈复杂度**：在某些判断中添加不必要的嵌套或冗余条件
- **添加防御性编程代码**：多余的null检查、不必要的类型转换
- **修改代码格式**：微调部分代码块的缩进、空行、括号位置

### 3.4 引入合理的代码瑕疵（关键但敏感）

> 这些瑕疵必须不影响代码的功能正确性

- **拼写错误**：仅在注释或字符串常量中，如 `recieve` 替代 `receive`，**绝不能出现在标识符中**
- **不完整注释**：`// 这里好像有问题，先这样`、`// TODO`
- **格式不一致**：部分用空格缩进，部分用Tab；部分运算符两边加空格，部分不加
- **遗漏的优化**：循环中重复计算某个值、未使用缓存
- **边界处理不完整**：模拟"没考虑到这种情况"
- **版本标记**：被注释掉的旧代码、`// v1.2 之前的逻辑，已废弃`

### 3.5 代码与注释中文化（硬性规范）

软著申请材料面向中国版权保护中心，代码中的提示信息与注释**必须中文化**。这也是去 AI 化的一部分——AI 生成的代码常默认输出英文 message 和英文注释，中文化能显著降低 AI 特征。

#### 规则1：message 消息内容中文化

所有面向用户的提示信息、异常文案、message 字段，若为英文必须改成中文：

| 场景 | ❌ 英文（违规） | ✅ 中文（达标） |
|------|----------------|----------------|
| 异常抛出 | `throw new BusinessException("User not found")` | `throw new BusinessException("用户不存在")` |
| 错误返回 | `return {code: 400, message: "Invalid params"}` | `return {code: 400, message: "参数错误"}` |
| 前端提示 | `wx.showToast({title: "Success"})` | `wx.showToast({title: "操作成功"})` |
| 校验提示 | `'email format error'` | `'邮箱格式错误'` |
| alert | `alert("Are you sure?")` | `alert("确定要删除吗？")` |

**例外（可保留英文）**：纯技术日志输出（如 `console.log('API response:', res)`）、HTTP 状态码描述、第三方库要求的固定字段值。

#### 规则2：注释中禁止英文（专用语除外）

注释必须用中文书写。允许保留英文的仅限：

- 函数名 / 类名 / 变量名（如 `// 调用 login() 方法`中的 `login`）
- API 名称、协议、技术术语（如 `// 通过 Redis 缓存`中的 `Redis`）
- 代码标识符引用

| ❌ 英文注释（违规） | ✅ 中文注释（达标） |
|--------------------|--------------------|
| `// check if user exists` | `// 检查用户是否存在` |
| `// TODO: fix this later` | `// TODO: 后面再修这里` |
| `// This function handles user auth` | `// 这个函数处理用户认证` |
| `// return 200 if success` | `// 成功返回 200` |

> 中文化与注释率 <5% 并行：删到只剩必要的中文注释即可，不要为了"凑中文"而增加注释量。

### 3.6 软著格式要求与去AI化的平衡

去AI化处理**必须**满足以下格式要求：

| 要求 | 具体规定 | 去AI化注意事项 |
|------|----------|----------------|
| 每页行数 | 不少于50行（删除空行后） | 行数不足时通过拆分链式调用、保留必要业务代码补充；**不可用注释凑行数**（注释率必须 <5%） |
| 页眉 | "软件全称+版本号" | 去AI化时不得破坏 |
| 页码 | 右上角连续编排 | 保持连续性 |
| 总代码量 | >3000行：前30+后30页（共60页）；<3000行：全部提交 | 不能为凑行数添加无意义代码 |
| **注释覆盖率** | **<5%（硬性指标）** | **删除一切显而易见的注释，仅保留复杂算法/业务规则/陷阱说明** |
| **message 中文化** | 提示/异常文案必须中文 | 英文 message 必须改中文（见 3.5 规则1） |
| **注释禁英文** | 注释必须中文（专用语除外） | 见 3.5 规则2 |
| 开头结尾 | 第1页第1行必须是模块开头（import/类定义）；第60页最后行必须是程序结尾 | 去AI化时不得破坏 |

## 四、用户手册去AI化策略

### 4.1 句式多样化

- **长短句交替**：将均匀的25-35字长句改造为长短不一
  - 原："本系统提供了完善的用户管理功能，支持用户的增删改查操作"
  - 改："用户管理？该有的都有。增、删、改、查，一个不落。"
- **引入不完整句**："登录成功后，就可以看到主界面了"、"点这里，然后选那个选项"
- **使用反问句和感叹句**："这个功能怎么用？很简单..."、"注意！这个操作不可撤销！"
- **打破"总-分-总"结构**：有些章节直接进入操作步骤，有些在中间插入注意事项
- **调整段落长度差异**：有些段落只有1-2句话，有些可以有10行以上

### 4.2 语言口语化与个性化

- **替换正式用语**：
  - "请单击此处以执行相应操作" → "点击这里就行了"
  - "系统将会提示用户进行确认" → "这时候会弹个框让你确认"
  - "本功能旨在提高工作效率" → "这个功能就是帮你省时间的"
- **添加口语化连接词**："话说回来"、"顺便说一句"、"不过要注意"、"其实很简单"替代AI常用的"此外"、"值得注意的是"
- **加入个人经验**："建议先备份数据，笔者曾经因为没备份丢过重要文件"、"这个功能我们内部测试的时候发现..."
- **多用第二人称**：用"你"而不是"用户"

### 4.3 内容组织人性化

- **添加注意事项和警告**：
  - `⚠️ 注意：这一步如果做错了，后面的数据会全部丢失！`
  - `💡 小贴士：按住Shift键可以多选，很多人不知道这个技巧`
- **承认局限性**："目前这个版本还不支持批量导出，后续会加入"、"由于浏览器兼容性原因，建议在Chrome下使用"
- **引用具体场景**："假设你是一个销售人员，需要给客户生成报价单..."替代抽象描述
- **添加版本变更说明**：标注"v2.0新增"、"v1.5之后改了逻辑"
- **截图标注人为化**：可以有轻微错位、手写字体风格箭头、不同颜色标注框

### 4.4 制造合理的人工痕迹

- **添加TODO标记**：
  - `（此处待补充详细截图）`
  - `（操作视频后续上传）`
  - `（v3.0版本将更新此部分说明）`
- **格式不一致**：不同章节的标题格式略有不同（"一、"和"1."混用），段落缩进略有差异
- **轻微的表达重复**：偶尔重复之前说过的内容，或用稍不同方式再次解释
- **个人化备注**："如果在使用过程中遇到问题，可以联系我们的技术支持（虽然响应可能有点慢，但一定会回复的）"
- **时间痕迹**：标注编写日期、最后更新日期、"截至2024年12月的版本"

### 4.5 用户手册格式要求与去AI化平衡

| 要求 | 规定 | 去AI化策略 |
|------|------|------------|
| 页数 | 前30+后30页（共60页），不足则全部 | 通过增加口语化说明、注意事项补充行数 |
| 每页行数 | 不少于30行（有图页面除外） | 同上 |
| 页眉页脚 | 软件名称+版本号，右上角页码 | 保持不变 |
| 内容要求 | 功能介绍、运行环境、安装部署、核心操作步骤 | 不减少必要功能说明，只改变表达方式 |
| 截图要求 | 软件名称/版本号一致，日期逻辑合理 | 可在截图标注上加入个人风格 |

## 五、综合去AI化工作流程

### 5.1 五阶段处理流程

**阶段一：AI初稿生成**
- 通过提示词设计，从一开始就引入"不完美"要求
- 代码提示词示例："生成一个用户管理模块的代码，注释不要太详细，变量命名简洁一些"
- 文档提示词示例："写一份用户手册，语言口语化一点，像给同事讲解一样"

**阶段二：结构级改造**
- 源代码：合并/拆分函数、调整代码组织顺序、增减注释、修改变量命名
- 用户手册：重组章节顺序、调整段落长度分布、增删过渡内容、插入TODO和版本说明

**阶段三：细节级改写（核心环节，需人工深度介入）**
- 源代码：逐行修改命名、添加个人化注释、引入代码瑕疵、调整格式一致性
- 用户手册：逐段改写，打破句式、口语化、添加个人经验、制造逻辑跳跃

**阶段四：一致性检查**
- 软件名称和版本号在源代码/文档中是否一致
- 总行数和页数是否满足要求
- 页眉页脚格式是否正确
- 截图信息与文字描述是否一致

**阶段五：最终润色**
- 以"普通审查员"视角审视：材料是否像真实程序员在正常工作状态下编写

### 5.2 不同策略效果对比

| 策略 | 操作复杂度 | AI检测率降幅 | 推荐度 | 适用场景 |
|------|------------|--------------|--------|----------|
| 同义词替换 | 低 | 20-25% | ★★☆☆☆ | 快速初步处理 |
| 句式调整 | 中 | 35-40% | ★★★☆☆ | 文本内容改写 |
| 口语化改写 | 中 | 55-60% | ★★★★☆ | 用户手册 |
| 注入个人风格 | 高 | 65-70% | ★★★★☆ | 全部材料 |
| 混合创作模式 | 高 | 80-85% | ★★★★★ | 核心材料 |
| 深度人工改写 | 极高 | 90-95% | ★★★★★ | 高价值软件 |

> **推荐策略**：对软著材料，采用**混合创作模式**——AI辅助生成初稿，人工深度改写注入个人特征，目标AI率<30%。

### 5.3 效率优化建议

- **批量处理**：使用编辑器查找替换处理重复模式（命名规范、注释风格）
- **模板化个人风格**：建立"个人风格库"收集常用口语化表达、注释模板、风格偏好
- **重点改造高风险部分**：优先改造AI特征最明显的部分（过长注释、过于规范命名、模板化段落开头/结尾）
- **利用工具辅助**：先用Prettier/ESLint统一格式，再手动引入不一致性
- **分阶段审核**：每完成一个阶段，用AI检测工具检测一次，及时调整策略

## 六、风险预警与合规底线

### 6.1 可以做 vs 绝对不能做

**可以做：**
- 改变语言表达风格（口语化、个性化）
- 调整代码格式和命名习惯
- 增减注释
- 引入合理的代码结构变化
- 添加个人经验和建议

**绝对不能做（红线）：**
- ❌ 虚构软件功能（软著材料必须真实反映软件功能）
- ❌ 编造不存在的代码逻辑（代码必须可以运行）
- ❌ 伪造开发日期和版本信息（必须与申请表一致）
- ❌ 抄袭他人代码（软著要求原创性，抄袭会导致驳回甚至法律风险）

### 6.2 实用建议

- **软件本身必须真实**：软著审查前提是软件真实存在且可运行
- **保留开发过程证据**：需求文档、设计文档、版本控制记录、测试记录——可证明软件由人工开发
- **核心模块亲自编写**：核心算法、关键业务逻辑等体现独创性的部分由开发人员亲自编写
- **AI辅助而非替代**：用AI生成框架/起草初稿，但核心业务逻辑、关键技术决策、个性化表达必须人工完成
- **适度去AI化**：不追求"零AI率"，目标<30%即可；最优秀的人工写作也可能显示10-20%的AI率
- **诚实为上策**：如审查机构询问，诚实回答——绝大多数地区不禁止AI辅助编写软著材料

## 八、核心要点总结

AI生成内容的检测技术主要基于**困惑度、突发性和统计特征分析**，反映AI内容"过于平滑、过于规范、过于完美"的本质。

去AI化的核心策略是：**打破规律性、注入随机性、引入不完美**——让AI内容在统计特征上更接近人类创作的"自然波动状态"。

- **源代码层面**：**注释率严格 <5%**（删除一切显而易见注释）、改变命名风格、引入代码瑕疵、调整结构一致性
- **用户手册层面**：句式多样化、口语化表达、内容个性化、制造人工痕迹

**最后的原则**：去AI化不是造假，而是让AI辅助材料更好地反映人工创作的真实状态。最好的去AI化不是让材料"看起来不像AI写的"，而是让材料"看起来就像一个真实的程序员写的"——带有个人的习惯、合理的瑕疵、以及那种只有人类才有的"不完美之美"。
