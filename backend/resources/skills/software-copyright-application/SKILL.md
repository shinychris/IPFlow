# 软件著作权申请材料编写助手

协助用户编写符合中国版权保护中心要求的软件著作权登记申请材料。

## 使用模式判断

首先判断用户处于哪种使用模式：

| 模式 | 判断依据 | 处理方式 |
|------|----------|----------|
| **模式一：材料转化** | 用户提供了源代码、项目文件、已有文档等材料 | 基于用户材料进行整理、格式化、补全 |
| **模式二：自由生成** | 用户仅提供软件想法、功能描述、基本信息，无实际材料 | 基于信息生成完整材料，或进一步询问后生成 |

**判断方法：**
- 用户上传了文件/代码/项目 → 模式一
- 用户仅文字描述软件想法/功能 → 模式二

---

## 模式一：材料转化模式

用户已提供材料，基于现有材料进行转化和整理。

### 工作流程

```
1. 分析用户提供的材料
   ├── 源代码文件 → 提取前30页+后30页，格式化
   ├── 项目文件 → 分析技术栈、功能模块
   └── 已有文档 → 检查格式，补充完善
2. 信息确认
   ├── 确认软件名称、版本号
   ├── 确认申请人信息
   └── 确认开发完成日期
3. 材料转化
   ├── 源代码文档：按格式要求整理
   ├── 软件说明书：基于代码功能编写
   └── 申请表信息：整理填写内容
4. 一致性检查
5. 输出最终材料
```

### 材料处理策略

**情况1：用户提供完整项目代码**
- 使用 `scripts/extract_source.py` 提取源代码
- 分析项目结构，识别技术栈和核心功能
- 基于代码编写软件说明书

**情况2：用户提供零散代码片段**
- 按逻辑顺序组织代码
- 补充必要的上下文说明
- 确保代码真实性和完整性

**情况3：用户已有部分文档**
- 检查现有文档格式是否符合要求
- 补充缺失的章节内容
- 统一软件名称、版本号等信息

### 信息确认清单

与用户确认以下信息（材料中已明确的可跳过）：

| 信息项 | 确认内容 |
|--------|----------|
| 软件全称 | 是否以"系统/平台/软件"结尾 |
| 版本号 | 是否为 V1.0 |
| 开发完成日期 | 具体日期 |
| 申请人 | 个人/企业名称 |
| 开发方式 | 独立/合作/委托开发 |

## 模式二：自由生成模式

用户未提供材料，需要根据软件想法生成完整申请材料。

### 信息评估与决策

首先评估用户提供的信息是否足够生成材料：

#### 信息充足标准（满足以下即可）

| 信息类型 | 最低要求 | 说明 |
|----------|----------|------|
| 软件名称 | ✅ 必须 | 软件叫什么 |
| 软件类型 | ✅ 必须 | 网站/APP/小程序/工具/系统等 |
| 主要功能 | ✅ 必须 | 软件能做什么（至少3-5个核心功能） |
| 目标用户 | 建议有 | 给谁用的 |
| 技术栈 | 建议有 | 用什么技术开发的 |
| 应用场景 | 可选 | 在什么场景下使用 |

#### 决策流程

```
用户提供软件想法
       ↓
评估信息充足性
       ↓
    ┌──┴──┐
   足够  不足
    ↓     ↓
 直接生成  询问补充信息
    ↑     ↓
    └── 用户不提供/无法提供
           ↓
        自由发挥生成
```

**判断规则：**
- **信息足够**：有软件名称、类型、3个以上核心功能 → 直接开始生成
- **信息不足**：缺少关键信息 → 询问用户补充
- **用户不补充**：基于已有信息合理推测，自由发挥生成完整材料

### 询问策略（信息不足时）

询问用户以下信息（每次询问2-3个关键问题）：

1. **软件基础信息**
   - 软件名称是什么？
   - 是什么类型的软件？（网站/APP/小程序/桌面软件/系统等）

2. **功能描述**
   - 软件主要有哪些功能？（请列举3-5个）
   - 解决了什么问题？

3. **技术与用户**
   - 目标用户是谁？
   - 使用什么技术开发的？（如知道的话）

**用户回应处理：**
- 用户提供了补充信息 → 更新信息评估，如足够则开始生成
- 用户表示无法提供更多 → 基于已有信息自由发挥生成
- 用户要求直接开始 → 基于已有信息自由发挥生成

### 自由发挥生成原则

当信息不足且用户不补充时，按以下原则生成：

1. **软件名称**：使用用户提供名称，如无则使用"XX智能管理系统"
2. **软件类型**：根据功能描述推断，如无法推断则默认为"Web管理系统"
3. **功能模块**：
   - 基于用户提供的核心功能扩展
   - 每个核心功能扩展为2-3个子功能
   - 添加通用的用户管理、数据管理等基础模块
4. **技术栈**：
   - 根据软件类型选择合理的通用技术栈
   - Web系统：Spring Boot + Vue.js + MySQL
   - APP：React Native / Flutter + 后端API
   - 小程序：微信小程序原生 / Taro
5. **代码生成**：
   - 生成符合常见开发规范的示例代码
   - 包含配置文件、实体类、业务逻辑等
   - 代码必须看起来像真实项目代码

## 申请材料生成规范

无论哪种模式，最终生成的材料必须符合以下规范：

### 1. 源代码文档

**格式要求：**
- 页数：前30页 + 后30页（共60页），不足则全部提交
- 行数：每页不少于50行（删除空行后）
- 页眉：软件名称 + 版本号
- 页码：右上角连续编排

**模式一处理：**
- 从用户代码中按逻辑顺序提取
- 保留用户代码的核心逻辑和结构
- 删除空行和敏感信息

**模式二生成：**
- 根据软件功能设计合理的代码结构
- 包含：配置类、实体类、业务逻辑、工具类等
- 代码风格统一，注释清晰

### 2. 软件说明书

**文档类型选择：**

| 软件类型 | 推荐文档 | 内容重点 |
|----------|----------|----------|
| 有界面应用 | 用户手册 | 界面截图位置、操作流程、功能说明 |
| 无界面系统 | 设计说明书 | 系统架构、技术方案、接口设计 |
| 算法/工具类 | 设计说明书 | 算法原理、输入输出、应用场景 |

**模式一处理：**
- 基于用户提供的代码/文档分析功能
- 如实描述已实现的功能
- 补充操作步骤说明

**模式二生成：**
- 根据软件功能设计完整的模块结构
- 编写详细的功能描述
- 提供合理的操作步骤说明（有界面软件）
- 提供系统架构设计（设计说明书）

### 3. 申请表信息

整理以下信息供用户在线填写：

| 字段 | 内容来源（模式一） | 内容来源（模式二） |
|------|-------------------|-------------------|
| 软件全称 | 用户确认 | 用户提供/生成 |
| 版本号 | V1.0 | V1.0 |
| 开发完成日期 | 用户确认 | 用户提供/合理推测 |
| 软件功能 | 从代码分析 | 从功能描述整理 |
| 技术特点 | 从代码分析 | 从技术栈整理 |
| 硬件环境 | 通用配置 | 通用配置 |
| 软件环境 | 从代码分析 | 从技术栈整理 |

---

## 一致性检查

生成完成后，检查以下一致性：

- [ ] 所有材料中的**软件名称**完全一致
- [ ] 所有材料中的**版本号**完全一致（V1.0）
- [ ] 源代码与说明书描述的**功能**相互对应
- [ ] **开发完成日期**逻辑合理
- [ ] 页眉页码格式统一

## 使用工具脚本

### 源代码提取脚本

从项目中提取符合格式要求的源代码：

```bash
python extract_source.py --project-path /path/to/project \
  --software-name "XX管理系统" \
  --version "V1.0" \
  --output-dir ./output
```

**适用场景：**
- 模式一：用户提供了项目代码，需要提取源代码文档
- 模式二：生成了示例代码，需要格式化成文档

## 重要提醒

1. **实名认证**：所有著作权人需在中国版权保护中心官网完成实名认证（约3个工作日）
2. **签章要求**：申请表需手写签名或加盖公章，不得使用电子章/PS
3. **代码真实性**：模式一下代码必须来自用户真实项目；模式二下生成的代码需符合真实开发规范
4. **材料一致性**：所有材料中的软件名称、版本号必须完全一致
5. **PDF格式**：所有材料最终需转换为PDF格式上传

## 申请流程概览

```
1. 官网注册 → 2. 实名认证 → 3. 准备材料 → 4. 在线填报 → 5. 提交审核 → 6. 等待下证
    ↑                                                              ↓
    └──────────────── 约3个工作日 ──────────────────────────────────┘
```

普通登记周期：约31-60个工作日（自受理之日起）
