# 软件著作权申请材料编写助手

协助用户编写符合中国版权保护中心要求的软件著作权登记申请材料。

## 使用模式判断

首先判断用户处于哪种使用模式：

| 模式 | 判断依据 | 处理方式 |
|------|----------|----------|
| **模式一：材料转化** | 用户提供了源代码、项目文件、已有文档等材料 | 基于用户材料进行整理、格式化、补全 |
| **模式二：自由生成** | 用户仅提供软件想法、功能描述、基本信息，无实际材料 | 基于信息生成完整材料，或进一步询问后生成 |

**判断方法：**
- 用户上传了文件/代码/项目 → 模式一
- 用户仅文字描述软件想法/功能 → 模式二

---

## 模式一：材料转化模式

用户已提供材料，基于现有材料进行转化和整理。

### 工作流程

```
1. 分析用户提供的材料
   ├── 源代码文件 → 提取前30页+后30页，格式化
   ├── 项目文件 → 分析技术栈、功能模块
   └── 已有文档 → 检查格式，补充完善
2. 信息确认
   ├── 确认软件名称、版本号
   ├── 确认申请人信息
   └── 确认开发完成日期
3. 材料转化
   ├── 源代码文档：按格式要求整理
   ├── 软件说明书：基于代码功能编写
   └── 申请表信息：整理填写内容
4. 一致性检查
5. 输出最终材料
```

### 材料处理策略

**情况1：用户提供完整项目代码**
- 使用 `scripts/extract_source.py` 提取源代码
- 分析项目结构，识别技术栈和核心功能
- 基于代码编写软件说明书

**情况2：用户提供零散代码片段**
- 按逻辑顺序组织代码
- 补充必要的上下文说明
- 确保代码真实性和完整性

**情况3：用户已有部分文档**
- 检查现有文档格式是否符合要求
- 补充缺失的章节内容
- 统一软件名称、版本号等信息

### 信息确认清单

与用户确认以下信息（材料中已明确的可跳过）：

| 信息项 | 确认内容 |
|--------|----------|
| 软件全称 | 是否以"系统/平台/软件"结尾 |
| 版本号 | 是否为 V1.0 |
| 开发完成日期 | 具体日期 |
| 申请人 | 个人/企业名称 |
| 开发方式 | 独立/合作/委托开发 |

## 模式二：自由生成模式

用户未提供材料，需要根据软件想法生成完整申请材料。

### 信息评估与决策

首先评估用户提供的信息是否足够生成材料：

#### 信息充足标准（满足以下即可）

| 信息类型 | 最低要求 | 说明 |
|----------|----------|------|
| 软件名称 | ✅ 必须 | 软件叫什么 |
| 软件类型 | ✅ 必须 | 网站/APP/小程序/工具/系统等 |
| 主要功能 | ✅ 必须 | 软件能做什么（至少3-5个核心功能） |
| 目标用户 | 建议有 | 给谁用的 |
| 技术栈 | 建议有 | 用什么技术开发的 |
| 应用场景 | 可选 | 在什么场景下使用 |

#### 决策流程

```
用户提供软件想法
       ↓
评估信息充足性
       ↓
    ┌──┴──┐
   足够  不足
    ↓     ↓
 直接生成  询问补充信息
    ↑     ↓
    └── 用户不提供/无法提供
           ↓
        自由发挥生成
```

**判断规则：**
- **信息足够**：有软件名称、类型、3个以上核心功能 → 直接开始生成
- **信息不足**：缺少关键信息 → 询问用户补充
- **用户不补充**：基于已有信息合理推测，自由发挥生成完整材料

### 询问策略（信息不足时）

询问用户以下信息（每次询问2-3个关键问题）：

1. **软件基础信息**
   - 软件名称是什么？
   - 是什么类型的软件？（网站/APP/小程序/桌面软件/系统等）

2. **功能描述**
   - 软件主要有哪些功能？（请列举3-5个）
   - 解决了什么问题？

3. **技术与用户**
   - 目标用户是谁？
   - 使用什么技术开发的？（如知道的话）

**用户回应处理：**
- 用户提供了补充信息 → 更新信息评估，如足够则开始生成
- 用户表示无法提供更多 → 基于已有信息自由发挥生成
- 用户要求直接开始 → 基于已有信息自由发挥生成

### 自由发挥生成原则

当信息不足且用户不补充时，按以下原则生成：

1. **软件名称**：使用用户提供名称，如无则使用"XX智能管理系统"
2. **软件类型**：根据功能描述推断，如无法推断则默认为"Web管理系统"
3. **功能模块**：
   - 基于用户提供的核心功能扩展
   - 每个核心功能扩展为2-3个子功能
   - 添加通用的用户管理、数据管理等基础模块
4. **技术栈**：
   - 根据软件类型选择合理的通用技术栈
   - Web系统：Spring Boot + Vue.js + MySQL
   - APP：React Native / Flutter + 后端API
   - 小程序：微信小程序原生 / Taro
5. **代码生成**：
   - 生成符合常见开发规范的示例代码
   - 包含配置文件、实体类、业务逻辑等
   - 代码必须看起来像真实项目代码

## 申请材料生成规范

无论哪种模式，最终生成的材料必须符合以下规范。

### ⚠️ 输出格式硬性要求

**所有文档（源代码文档、用户手册/设计说明书）必须输出为 `.docx` 格式。**

- 使用 `python-docx` 库生成（详见 `scripts/extract_source.py`）
- 不再支持 txt、md 作为最终交付格式（txt/md 仅作中间草稿）
- 用户最终提交给中国版权保护中心时，再将 docx 转为 PDF（中心要求 PDF 上传）

### 软著申请 docx 文档规范

#### A. 通用规范（源代码 docx + 用户手册 docx 共用）

| 项 | 规范 | 说明 |
|----|------|------|
| **纸张** | **A4（210×297mm）** | 必须设置为 A4，不得使用 Letter |
| **页边距** | 常规（上下 1"，左右 0.8–1.25"） | 即上下 2.54cm，左右约 2.0–3.17cm |
| **打印** | 单面 | （PDF 阶段） |
| **页眉** | **所有页面显示软件全称 + 版本号**（如"销售通智能销售助手软件 V1.0"） | 居中，小五号字；缺失会被补正 |
| **页码** | **连续页码**（多文件合并时 1–30 / 31–60 连续） | 位置：底部居中（项目文档可右上角） |
| **软件名称一致性** | 所有材料中软件全称、版本号一字不差 | 申请表 = 源代码页眉 = 说明书页眉 |
| **字号（正文）** | 小四（12pt）宋体；标题黑体 | 仅适用于用户手册/说明书 |

#### B. 代码文件特有规范（仅源代码 docx）

| 项 | 规范 |
|----|------|
| **字体** | **Consolas / Courier New（等宽字体）**，不得使用宋体 |
| **字号** | 五号（10.5pt） |
| **行距** | 单倍 |
| **每页代码量** | **≥ 50 行（不含空行）** |
| **内容范围** | 真实源代码（非空行装饰） |
| **文件标识** | 每个源文件前注明文件路径（如「代码文件: src/main/UserService.java」） |
| **前后端组织** | 推荐分区呈现（后端 / 前端） |
| **提交页数** | 前 30 页 + 后 30 页，不足 60 页全交 |
| **注释率** | **<5%（硬性，详见"AI痕迹去除"）** |
| **message 中文化** | 代码中 message/提示信息/异常文案若为英文，**必须改成中文**（如 `'User not found'` → `'用户不存在'`） |
| **注释禁英文** | 注释中除函数名/类名/专业术语外，**禁止出现英文**，必须用中文 |

### ⚠️ 代码中文化与注释规范

软著申请材料面向中国版权保护中心，代码与注释**必须中文化**：

**规则1：message 消息内容中文化**
- 所有 `throw new XxxException("...")`、`message: "..."`、`alert("...")`、`console.log("...")`、`wx.showToast({title: "..."})` 中的提示文案，若为英文必须改中文
- ✅ `throw new BusinessException("用户不存在")`
- ❌ `throw new BusinessException("User not found")`
- ✅ `message: "登录成功"`
- ❌ `message: "Login success"`
- 例外：日志输出中的技术信息（如 `console.log('API response:', res)`）可保留英文，但面向用户的提示必须中文

**规则2：注释中禁止英文（专用语除外）**
- 注释必须用中文书写
- 允许保留英文的：函数名、类名、变量名、API 名称、技术术语（如 `// 调用 login() 接口` 中的 `login`）
- ❌ `// check if user exists` → ✅ `// 检查用户是否存在`
- ❌ `// TODO: fix this later` → ✅ `// TODO: 后面再修这里`
- ❌ `// This function handles user auth` → ✅ `// 这个函数处理用户认证`

### 1. 源代码文档

**生成要求：**
- 输出格式：**.docx**（A4 纸张，Consolas 字体，五号，单倍行距）
- 页数：前 30 页 + 后 30 页（共 60 页），不足则全部提交
- 行数：每页不少于 50 行（删除空行后）
- 页眉：软件全称 + 版本号（居中）
- 页码：连续编排（底部居中）
- 文件标识：每个源文件前注明「代码文件: <相对路径>」

**模式一处理：**
- 使用 `scripts/extract_source.py` 从用户项目提取
- 保留用户代码的核心逻辑和结构
- 删除空行和敏感信息
- 注释率压到 5% 以下

**模式二生成：**
- 根据软件功能设计合理的代码结构
- 包含：配置类、实体类、业务逻辑、工具类等
- 代码风格统一，**注释率严格控制在 5% 以下**（详见"AI痕迹去除"章节）

### 2. 软件说明书

**输出格式：.docx**（A4 纸张，正文小四宋体，标题黑体，1.5 倍行距，首行缩进 2 字符）

**文档类型选择：**

| 软件类型 | 推荐文档 | 内容重点 |
|----------|----------|----------|
| 有界面应用 | 用户手册 | 界面截图位置、操作流程、功能说明 |
| 无界面系统 | 设计说明书 | 系统架构、技术方案、接口设计 |
| 算法/工具类 | 设计说明书 | 算法原理、输入输出、应用场景 |

**模式一处理：**
- 基于用户提供的代码/文档分析功能
- 如实描述已实现的功能
- 补充操作步骤说明

**模式二生成：**
- 根据软件功能设计完整的模块结构
- 编写详细的功能描述
- 提供合理的操作步骤说明（有界面软件）
- 提供系统架构设计（设计说明书）

### 3. 申请表信息

整理以下信息供用户在线填写：

| 字段 | 内容来源（模式一） | 内容来源（模式二） |
|------|-------------------|-------------------|
| 软件全称 | 用户确认 | 用户提供/生成 |
| 版本号 | V1.0 | V1.0 |
| 开发完成日期 | 用户确认 | 用户提供/合理推测 |
| 软件功能 | 从代码分析 | 从功能描述整理 |
| 技术特点 | 从代码分析 | 从技术栈整理 |
| 硬件环境 | 通用配置 | 通用配置 |
| 软件环境 | 从代码分析 | 从技术栈整理 |

## AI痕迹去除（核心增强）

软著申请材料**必须**经过 AI 痕迹去除处理。详细策略参见 [references/anti-ai-detection.md](references/anti-ai-detection.md)。

**核心思路**：打破规律性、注入随机性、引入不完美

| 改造对象 | 关键操作 | 目标特征 |
|----------|----------|----------|
| **源代码** | **注释率硬性要求 <5%**（删除一切显而易见的注释，仅保留复杂算法/关键业务规则的极简说明）、混用命名缩写（`userAuthenticationToken`→`uauthToken`）、添加 TODO/FIXME/HACK 标记、保留合理瑕疵（注释拼写错误、遗漏优化、版本痕迹）、删除 JSDoc/JavaDoc 等模板化块注释 | **注释覆盖率 <5%**，函数长度不均匀 |
| **用户手册** | 长短句交替、口语化（"点击这里就行了"）、添加 💡小贴士/⚠️注意、引入个人经验、TODO 标记、第二人称"你" | 句长方差大、连接词密度 <9%、段落长度不一 |

### ⚠️ 注释率硬性指标（源代码）

- **目标**：注释行数 / 代码总行数 **< 5%**
- **核查工具**：`python scripts/ai_trace_analyzer.py --type code --path <文件>`
- **执行原则**：
  - 默认删除所有"显而易见"的注释（如 `i++ // 自增`、`// 用户登录方法`）
  - 仅保留：复杂算法解释、关键业务规则、防止后续误改的"陷阱注释"
  - 严禁使用统一的 JSDoc/JavaDoc 块注释模板（这是 AI 代码最强指纹）
  - 保留的注释应风格不一：有的单行 `//`，有的 `# TODO`，有的干脆不写

**红线（不可触碰）**：❌ 虚构软件功能 ❌ 编造无法运行的代码 ❌ 伪造开发日期/版本号 ❌ 抄袭他人代码。去AI化是让 AI 辅助材料"看起来像人工写的"，不是伪造材料。

### AI 痕迹自检脚本

使用 `scripts/ai_trace_analyzer.py` 评估材料的关键 AI 特征指标（注释覆盖率、句长方差、连接词密度、词汇多样性 TTR 等）：

```bash
python scripts/ai_trace_analyzer.py --type code --path ./source-code-front.md
python scripts/ai_trace_analyzer.py --type text --path ./user-manual.md
python scripts/ai_trace_analyzer.py --type text --path ./materials/ --batch
```

输出 AI 风险评分（0-100）、识别到的 AI 特征、改进建议。**目标**：风险评分 <30（低风险，接近人类特征），且**源代码注释覆盖率必须 <5%**（硬性指标，不达标必须返回修改）、**英文 message 数 = 0、英文注释行数 = 0**（中文化硬性规范）。

## 一致性检查

生成完成后，检查以下一致性：

- [ ] 所有材料中的**软件名称**完全一致
- [ ] 所有材料中的**版本号**完全一致（V1.0）
- [ ] 源代码与说明书描述的**功能**相互对应
- [ ] **开发完成日期**逻辑合理
- [ ] 页眉页码格式统一

## 使用工具脚本

### 源代码提取脚本（生成 .docx）

从项目中提取并生成符合 docx 规范的源代码文档（A4、Consolas、五号、单倍行距、页眉软件名）：

```bash
python scripts/extract_source.py --project-path /path/to/project \
  --software-name "XX管理系统" \
  --version "V1.0" \
  --output-dir ./output
```

> 脚本依赖 `python-docx`：`pip install python-docx`。脚本仅输出 `.docx`，不再支持 txt。

**适用场景：**
- 模式一：用户提供了项目代码，需要提取源代码文档
- 模式二：生成了示例代码，需要格式化成文档

### AI 痕迹自检脚本

```bash
python scripts/ai_trace_analyzer.py --type code --path ./output/源代码_前30页_xxx.docx
python scripts/ai_trace_analyzer.py --type text --path ./output/用户手册_xxx.docx
```

> 源代码注释率必须 <5%，英文 message 数 = 0、英文注释行数 = 0，AI 风险评分 <30 才可交付。
