# IPFlow 上线前模拟用户测试报告

> **被测对象**：IPFlow 知识产权助手 v2.0（后端 FastAPI，`/api/v1`，100 路由）
> **测试形态**：后端 API 真实驱动（`sim_run.py` SQLite 仿真模式，无外部 Postgres/Redis/MinIO）
> **测试时间**：2026-06-30
> **测试人**：AI 仿真测试（simulated-usability-testing 技能）
> **结论预告**：🟡 **修复后上线** — 核心业务链路全通、产物真实可下载，但发现 3 项严重缺陷须修复后方可上线。

| 通过准则 | 结果 |

## 五、发现的问题（缺陷清单）

### 🔴 严重（serious）— 阻塞上线，必须修复

#### 缺陷 #3 【严重】软著导出源代码换行符被转义为字面 `\n`
- **现象**：`02_源代码_鉴别材料.txt` 中行间分隔符为字面 `\n`（字节 `5c 6e`）而非真实换行（`0a`）；整个源代码实际呈单行文本，破坏软著「每页50行带行号、连续打印」的强制规范。
- **影响**：软著申请的**核心交付物**（源代码鉴别材料）不可用，直接影响用户材料能否通过版权中心审查。
- **根因**：`backend/src/ipflow/services/copyright/code_processor.py`
  - L402 `separator = f"\\n// ====..."`（应为 `\n`）
  - L443 / L457 / L472 `page_content = '\\n'.join(...)`（应为 `'\n'.join(...)`）
  - 文件分隔符与页面内容连接全部误用双反斜杠转义；`pages_data` 入库即含字面 `\n`，导出原样写出。
- **复现**：上传任意代码 ZIP → 导出 → 解压 `02_源代码_鉴别材料.txt` → 用 `xxd` 见 `\n` 为 `5c 6e`。
- **修复建议**：将上述 4 处 `'\\n'` 改为 `'\n'`；加单测断言导出文本含真实换行（`chr(10)`）。

#### 缺陷 #5 【严重】尼斯分类全局查询路由被错误挂到 `projects/{project_id}` 前缀
- **现象**：`GET /api/v1/trademarks/nice-classes`（用户创建商标前浏览 45 类的入口）返回 **404**。
- **影响**：用户在创建商标项目**之前**无法浏览尼斯分类，严重影响商标申请流程的易用性与功能完整性。
- **根因**：`backend/src/ipflow/api/v1/trademarks/__init__.py:16` 把整个 `nice_classification_router`（含全局 `/nice-classes`、`/nice-classes/by-id/{id}` 与项目级 `/nice-classes/selected`）统一挂到 `prefix="/projects/{project_id}"` 下，导致全局端点需带 `project_id`。
- **复现**：`curl /api/v1/trademarks/nice-classes` → 404；`curl /api/v1/trademarks/projects/{id}/nice-classes` → 200（45 类数据本身正确）。
- **修复建议**：将 `/nice-classes` 与 `/nice-classes/by-id/{id}` 拆到无 project 前缀的路由；或前端用带 project_id 的路径（不推荐，破坏 REST 语义）。

#### 缺陷 #6 【严重】GET `/health` 完整健康检查返回 500
- **现象**：`GET /health` → `{"success":false,"code":"INTERNAL_ERROR"}`，ImportError。
- **影响**：运维/监控系统（K8s 探针、告警）无法做依赖探测；`/health/simple` 可用，不影响业务流量。
- **根因**：`backend/src/ipflow/main.py:128` `from ipflow.db import async_session_maker`，但 `db/session.py` 仅暴露 `AsyncSessionLocal`（模块名不匹配）。
- **修复建议**：改为 `from ipflow.db import AsyncSessionLocal as async_session_maker` 或直接用 `async_session_maker` 实际导出名；加启动冒烟测试。

### 🟡 轻微（minor）

- **缺陷 #1**：重复注册同邮箱返回 **422 VALIDATION_ERROR**（msg=该邮箱已被注册）而非 REST 惯例的 409/400。语义正确（已拦截），仅状态码不规范。建议统一为 409。
- **缺陷 #2**：说明书 `page_count`/`word_count` 由服务端按 `content_html` 纯文本**重算**（`manuals.py:324-325`：`word_count=len(text)`, `page_count=max(1,word_count//800)`），忽略客户端提交值。属**防伪造的设计决策**，非 bug，但应在 API 文档明确说明（避免前端困惑）。

### 🟢 建议（suggestion）

- **发现 #4**：专利与商标导出在 `python-docx` 可用时**同时生成 DOCX 版本**（超出 PRD 的 ZIP+TXT 要求），文档化处理完善，建议在 PRD/功能清单中补录该能力。

## 六、测试范围与局限

### 已充分覆盖（强证据）
- ✅ 认证（注册/登录/鉴权/异常）
- ✅ 项目管理（CRUD/隔离/类型不匹配拦截）
- ✅ 软著全链路（软件信息/代码处理/说明书/合规/导出/下载，含真实产物）
- ✅ 专利全链路（专利信息/权利要求/从属校验/导出/下载，含真实产物）
- ✅ 商标全链路（商标信息/尼斯分类/选择/导出/下载，含真实产物）
- ✅ 合规检查（规则明细、`can_export` 门控正确）
- ✅ 数据校验（完整性/一致性/计算/隔离，11 项）

### 未覆盖 / 局限（诚实标注）
- ⚪ **前端 UI（Next.js）**：本次为后端 API 直测，未做浏览器渲染、表单交互、视觉可用性。建议另起一轮浏览器仿真（`browser_eval`）。
- ⚪ **配套能力**：组织/成员/邀请、订阅/计划、后台/审计、Stripe 真实支付、Celery 异步队列 —— 仅冒烟，未深入（PRD 标注为可选配套）。
- ⚪ **直连数据库批量校验**：本次数据校验基于 API 响应/网络请求/导出产物（仿真可见数据），未做跨表批量 SQL 校验（交由 software-testing 沉淀）。
- ⚪ **AI 草稿生成（claude_code/agent 后端）**：本次用默认 `template` 后端（无 LLM）；`claude_code`/`agent` 后端需专用部署机，未测。
- ⚪ **并发/超时**：未做大规模并发与超卖/重复提交压测。
- ⚪ **环境差异**：sim 模式用 SQLite，生产用 PostgreSQL；SQLite 适配（UUID/外键）已验证，但生产 PG 的个别行为（如并发、JSON 字段）未在此模式覆盖。
