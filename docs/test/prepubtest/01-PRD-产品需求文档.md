# IPFlow 上线前测试 — 产品需求文档（PRD）

> **用途**：本次为「上线前完整功能测试」的依据文档。PRD 描述产品应当做什么，作为编写用户故事/测试用例/上线结论判断的基准。
> **被测对象**：IPFlow 知识产权助手 v2.0（后端 FastAPI 服务，`/api/v1`，共 100 个路由）
> **测试形态**：后端 API 真实驱动（`sim_run.py` SQLite 仿真模式，无需外部 Postgres/Redis/MinIO）
> **日期**：2026-06-30

---

## 一、产品定位

IPFlow 是面向**中国知识产权申报材料准备**场景的辅助工具，帮助用户高效**生成、整理、核对、导出**三类申报材料：**软件著作权（软著）、专利、商标**。

- 它是**材料辅助工具**，不是完整的申请管理平台、代理业务系统或强 SaaS 商业平台。
- 组织、订阅、后台等模块为当前代码保留的**配套能力**，可按部署选择启用，本次不作为上线核心把关项。

## 二、目标用户与角色

| 角色 | 典型场景 | 本次测试覆盖 |
|------|---------|------------|
| 独立开发者 / 个人 | 为自己的项目申请软著 | ✅ 主测（注册新用户） |
| 科技企业 / 机构 | 批量整理知识产权材料 | ✅（同一账号多项目） |
| 知识产权服务机构 | 协助客户准备材料 | ✅（同上） |
| 组织管理员（配套） | 组织内成员协作 | ⚪ 配套，不作为核心把关 |

## 三、核心功能需求（上线核心把关项）

### 3.1 认证（Auth）
- **R-AUTH-1**：访客可注册账号（邮箱+用户名+密码），注册成功即登录（返回 access_token）。
- **R-AUTH-2**：可用用户名或邮箱+密码登录，返回 access/refresh token。
- **R-AUTH-3**：受保护端点需 `Authorization: Bearer <token>`，token 失效/缺失应 401。
- **R-AUTH-4**：邮箱验证为**非阻断**（发送失败不阻塞注册/登录）。

### 3.2 项目管理（Projects）
- **R-PROJ-1**：可创建项目，指定 `project_type`（copyright/patent/trademark）与 `subject_type`。
- **R-PROJ-2**：项目按 `owner_id` 隔离，用户只能看到/操作自己的项目。
- **R-PROJ-3**：可列表、查询、更新、软删除、复制项目。

### 3.3 软著申请（Copyright）— 5 步向导
- **R-CR-1（软件信息）**：可填写/更新软件信息（全称、简称、版本、开发语言、功能描述、技术特点等）。
- **R-CR-2（代码材料）**：可上传源代码 ZIP，系统自动**识别代码文件、合并排序、生成前30页+后30页（每页50行，带行号页眉）**，并统计总行数/文件数/语言分布。
- **R-CR-3（说明书）**：可创建/更新操作说明书（模板类型 web/mobile/algorithm/script/desktop，标题+正文）。
- **R-CR-4（合规检查）**：可对项目运行合规检查，返回规则明细与 `can_export` 结论。
- **R-CR-5（导出）**：合规通过后可**生成导出 ZIP 并下载**，ZIP 内含：01_软件信息、02_源代码鉴别材料（前30+后30）、03_操作说明书、04_材料清单、05_打印指南、06_网报对照表。

### 3.4 专利申请（Patent）— 5 步向导
- **R-PT-1（专利信息）**：可填写专利信息（类型 invention/utility_model/design、标题、技术领域、背景技术、摘要）。
- **R-PT-2（权利要求）**：可添加权利要求（独立/从属，从属需引用更小编号）。
- **R-PT-3（说明书）**：发明内容/具体实施方式等。
- **R-PT-4（导出）**：可**生成导出 ZIP 并下载**，含：说明书 + 权利要求书。

### 3.5 商标申请（Trademark）— 4 步向导
- **R-TM-1（商标信息）**：可填写商标信息（类型 word/device/composite/3d/sound/color、名称、描述、颜色说明、图样说明）。
- **R-TM-2（尼斯分类）**：可查询 45 类尼斯分类，并为商标**选择分类+商品服务项**。
- **R-TM-3（导出）**：可**生成导出 ZIP 并下载**，含：商标注册申请书（含分类清单）。

### 3.6 配套能力（非核心把关）
- **R-SUP-1**：组织/成员/邀请、订阅/计划、后台/审计、支付/Stripe —— 代码保留，本次仅做冒烟（不阻塞）。

## 四、非功能需求（上线观察项）

| 维度 | 要求 | 优先级 |
|------|------|-------|
| 安全 | 密码 bcrypt、JWT、按 owner_id 隔离、上传安全校验 | P0 |
| 健壮性 | 外部依赖（Redis/MinIO/SMTP）不可用时优雅降级，不崩主流程 | P0 |
| 数据正确 | 代码行数/页数/语言统计、导出内容与填写内容一致 | P0 |
| 文档 | `/docs`（OpenAPI）可访问 | P1 |

## 五、上线验收标准（本次测试判定基准）

1. **P0 用例全绿**：三业务核心路径（创建→填写→生成→合规→导出→下载）全部可走通，且能**真实下载到生成的结果件**。
2. **无致命/严重缺陷**：无 500 崩溃主流程、无数据丢失、无越权。
3. **无核心数据错误**：代码行数/页数统计、导出内容一致性、状态流转正确。
4. **关键用户故事已覆盖**：软著/专利/商标各至少 1 条端到端 happy path。
5. 上述满足 → 🟢 可上线；有严重/数据错 → 🔴/🟡。

## 六、本次测试边界

- ✅ **在范围内**：后端 API 全链路（auth→projects→三业务→合规→导出→下载），含数据校验与真实产物生成。
- ⚪ **配套/不深入**：前端 UI（Next.js）、Stripe 真实支付、组织协作、Celery 异步队列。
- ❌ **不在范围**：直连数据库批量校验、前端浏览器渲染截图（本次后端 API 直测）。

> 说明：本次采用「后端 API 直测」而非浏览器 UI，是因为核心目标是验证**数据输入与结果生成的正确性**，API 直测能最直接读到接口响应与持久化数据，符合「完整测试（含数据）」要求。前端 UI 的可用性建议另起一轮浏览器仿真。
