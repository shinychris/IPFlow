# IPFlow 发布与回滚演练手册

## 目标

建立可执行的 `staging -> canary -> production` 发布策略，并提供应用层与数据库层双维度回滚步骤。

## 发布前门禁

- 后端测试：`pytest -m "unit or integration"` 全绿
- 前端质量：`npm run check && npm run lint`
- E2E 烟测：`npm run test:e2e`
- 迁移验证：`alembic upgrade head`（空库 + 升级链路）

## 分阶段发布流程

### 1) Staging

1. 部署后端与前端镜像到 staging 环境。
2. 执行数据库迁移：`alembic upgrade head`。
3. 执行全链路验收：认证、建项、三业务编辑、导出、支付 mock/sandbox 回调。
4. 记录基线指标：错误率、P95 延迟、支付成功率。

### 2) Canary（10% 流量）

1. 仅放量 10% 用户流量到新版本。
2. 观察 30-60 分钟核心指标：
   - 5xx 比例
   - 登录成功率
   - 项目创建成功率
   - 支付状态回调成功率
3. 指标稳定后提升到 50%，再次观察。

### 3) Production 全量

1. 将流量切换到 100%。
2. 持续观察 2 小时。
3. 宣布发布完成并归档发布记录。

## 回滚演练（必须每次发布前验证）

### A. 应用层回滚

1. 将流量切回上一个稳定版本镜像（tag 回退）。
2. 验证健康检查与关键 API。
3. 确认前端入口与核心页面正常。

### B. 数据库层回滚

1. 对可逆迁移执行 `alembic downgrade -1` 演练。
2. 对不可逆迁移执行“备份恢复”演练：
   - 发布前做逻辑备份
   - 回滚时恢复到快照并重放必要事件
3. 验证数据一致性（支付订单、订阅状态、项目数据）。

## 事故响应与止损

- 任一关键指标连续 5 分钟恶化（例如 5xx > 1%）立即触发回滚。
- 回滚后进入故障分析流程（RCA），冻结继续发布。
- 修复后重新经过 staging 与 canary。

## 生产密钥切换（mock_first -> production）

1. staging 使用 sandbox key 验证闭环。
2. 通过环境变量切换到生产 key（不改代码路径）。
3. canary 阶段只对小流量启用生产 key。
4. 指标稳定后全量启用。
