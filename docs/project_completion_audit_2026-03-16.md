## 3. 分模块完成度评估

## 3.1 后端（FastAPI + SQLModel）

**整体评价：较高完成度，但存在关键可用性风险点。**

已具备：

- 应用入口、路由聚合、健康检查、核心业务模块（auth/projects/copyright/patents/trademarks/subscriptions/admin/ai）
- 专利/商标/软著三大业务线的基础 CRUD 与导出接口框架
- 组织、订阅、管理后台等商业化 API 框架

主要问题：

- `payments` 路由未接入 `api_router`（在聚合文件中被注释）
- `subscriptions.py` 存在函数名遮蔽导致的递归调用风险（`get_usage_stats`、`handle_webhook`）
- `payments.py` 中存在同步/异步和类型不一致风险（`Session` 未正确导入/使用风格混杂）
- 支付签名校验与部分支付逻辑仍是 TODO/占位实现

结论：**业务主链路多数已实现，但支付和订阅回调链路还不够“生产可用”。**

## 3.2 前端（Next.js App Router + Zustand）

**整体评价：功能面覆盖较广，但迁移中有明显遗留。**

已具备：

- 认证、仪表盘、项目、软著/专利/商标页面框架
- 组件体系（含大量 UI 组件）与状态管理（Zustand）基础
- API 客户端与拦截器机制

主要问题：

- 存在路径拼接不一致：`apiClient` 已含 `/api/v1`，`ai-config.ts` 仍调用 `/api/v1/...`，可能导致双前缀请求
- 个别页面存在明显逻辑瑕疵（例如使用 `useState` 处理副作用同步而非 `useEffect`）
- 仍可见旧架构风格残留（`queryClient` fetch 模式与新 API 客户端混用）
- 无前端自动化测试文件，难以验证回归质量

结论：**页面和交互基础较完整，但一致性与稳定性仍需收敛。**

## 3.3 数据模型与迁移

**整体评价：模型扩展快于迁移，存在一致性隐患。**

- 模型已包含 `patent.py`、`trademark.py`、`payment.py` 等扩展实体
- Alembic 版本文件仅看到 2 个历史迁移，和当前模型规模可能不匹配
- 支付模型中外键/表名约定与现有用户表命名存在潜在不一致风险（需通过实际迁移验证）

结论：**模型层看起来“有”，但数据库演进闭环需要补强。**

## 3.4 测试与质量保障

**整体评价：后端有基础，前端薄弱。**

- 后端有 unit + integration 测试目录与样例
- 前端无测试文件，缺失自动化回归保障
- 当前报告为静态审计，未执行 pytest/tsc/build 进行运行态验证

结论：**测试体系尚未达到可发布级别。**

## 5. 风险分级（按上线影响）

### 高风险（建议优先修复）

- 支付模块未接入主路由，支付功能对外不可用或不可达
- 订阅模块中的递归调用风险可能导致运行时错误
- 代码库处于大规模迁移变更状态，当前基线不稳定

### 中风险

- 前后端 API 路径约定存在局部不一致（影响 AI 配置功能可用性）
- 数据模型与迁移文件数量不匹配，数据库升级风险较高
- 文档与实现不一致，容易造成联调误判

### 低风险

- 多处 TODO/占位实现集中在增强功能，不阻断核心 CRUD，但会影响商业化闭环
