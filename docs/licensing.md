# IPFlow 许可与商业化说明

> 本文档说明 IPFlow 的开源许可证（MIT）与商业化部署之间的关系，帮助团队明确授权边界。
> 最后更新：2026 年 6 月 30 日

---

## 一、当前许可证：MIT

仓库根目录的 `LICENSE` 采用 **MIT License**，这是宽松的开源许可证，核心含义：

| 权利 | 说明 |
|------|------|
| ✅ 商业使用 | 允许将代码用于商业目的 |
| ✅ 修改 | 允许修改源码 |
| ✅ 分发 | 允许分发原版或修改版 |
| ✅ 私有化使用 | 允许闭源使用（无需公开你的改动） |
| ⚠️ 唯一义务 | **分发时必须保留原 LICENSE 与版权声明** |

## 二、MIT 与"上架商业化"的关系

**MIT 允许商业使用，但"开源 MIT"和"商业化 SaaS 商品"是两件事：**

- **MIT 解决的是"代码授权"**：你可以合法地把这份代码部署成 SaaS 并收费。
- **商业化还需要"产品化能力"**：安全加固、支付闭环、合规法务、运维监控等，这些是评估报告中指出的 P0/P1 缺口。

换句话说：**MIT 许可证不阻碍商业化，但代码本身的就绪度才是商业化门槛。**

## 三、商业化部署的授权建议

### 方案 A：保持 MIT，自建 SaaS（推荐起步）

- 直接基于本仓库部署，加你自己的品牌、域名、支付配置
- **必须**：在前端 footer 与发行物中保留原 MIT 版权声明（`Copyright (c) 2024 IPFlow Team`）
- 适合：小团队快速验证商业模式

### 方案 B：双许可证（Dual Licensing，适合规模化）

当业务规模化后，建议采用**双许可**模式，平衡开源与商业：

- **社区版**：MIT 开源（保持当前仓库）
- **企业版 / 云服务版**：专有许可证（专有条款限制再分发，或仅以 SaaS 形式提供）

做法：
1. 新增专有许可证文件（如 `LICENSE-COMMERCIAL`）
2. 企业版功能（如高级 AI、SSO、审计）放私有仓库或单独分支
3. 贡献者协议（CLA）确保你有权将社区贡献转为专有许可

### 方案 C：完全闭源（需替换 LICENSE）

如决定闭源商业化：
- 将 `LICENSE` 从 MIT 改为专有许可证（如 BSUL、EULA）
- **注意**：MIT 历史版本已永久开源，已分发的代码无法回收授权；闭源只对"新代码 + 新发行版"生效
- 强烈建议保留社区开源版作为获客入口

## 四、第三方依赖许可证审查

商业化部署前，需确认所有依赖的许可证与商用兼容。当前主要依赖：

| 依赖 | 许可证 | 商用兼容 |
|------|--------|---------|
| FastAPI | MIT | ✅ |
| Next.js / React | MIT | ✅ |
| PostgreSQL driver (asyncpg) | Apache-2.0 | ✅ |
| MinIO SDK | Apache-2.0 | ✅ |
| Stripe SDK | MIT | ✅ |
| pydantic-ai | MIT | ✅ |
| shadcn/ui | MIT | ✅ |

> 建议在 CI 中接入 `pip-licenses`（后端）与 `license-checker`（前端）自动扫描依赖许可证，及时识别 GPL/AGPL 等强 copyleft 风险。

```bash
# 后端依赖许可证扫描
uv pip list --format=json | uv run pip-licenses --from=json --format=table

# 前端依赖许可证扫描
pnpm exec license-checker --summary
```
